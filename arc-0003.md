<h1><a id="user-content-algorand-standard-asset-parameters-conventions-for-fungible-and-non-fungible-tokens" class="anchor" aria-hidden="true" href="#algorand-standard-asset-parameters-conventions-for-fungible-and-non-fungible-tokens"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Algorand Standard Asset Parameters Conventions for Fungible and Non-Fungible Tokens</h1>
<h2><a id="user-content-summary" class="anchor" aria-hidden="true" href="#summary"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Summary</h2>
<p>This document introduce conventions for the parameters of Algorand Standard Assets (ASAs) for fungible tokens and non-fungible tokens (NFTs).</p>
<h2><a id="user-content-abstract" class="anchor" aria-hidden="true" href="#abstract"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Abstract</h2>
<p>The goal of these conventions is to make it simpler for block explorers, wallets, exchanges, marketplaces, and more generally, client software to display the properties of a given ASA.</p>
<h2><a id="user-content-specification" class="anchor" aria-hidden="true" href="#specification"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Specification</h2>
<p>The key words "<strong>MUST</strong>", "<strong>MUST NOT</strong>", "<strong>REQUIRED</strong>", "<strong>SHALL</strong>", "<strong>SHALL NOT</strong>", "<strong>SHOULD</strong>", "<strong>SHOULD NOT</strong>", "<strong>RECOMMENDED</strong>", "<strong>MAY</strong>", and "<strong>OPTIONAL</strong>" in this document are to be interpreted as described in <a href="https://www.ietf.org/rfc/rfc2119.txt" rel="nofollow">RFC-2119</a>.</p>
<blockquote>
<p>Comments like this are non-normative.</p>
</blockquote>
<p>An ARC-3 ASA has an associated JSON Metadata file, formatted as specified below, that is stored off-chain.</p>
<h3><a id="user-content-asa-parameters-conventions" class="anchor" aria-hidden="true" href="#asa-parameters-conventions"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>ASA Parameters Conventions</h3>
<p>The ASA parameters should follow the following conventions:</p>
<ul>
<li><em>Unit Name</em> (<code>un</code>): no restriction but <strong>SHOULD</strong> be related to the name in the JSON Metadata file</li>
<li><em>Asset Name</em> (<code>an</code>): <strong>MUST</strong> be:
<ul>
<li>either exactly <code>arc3</code> (without any space)</li>
<li>or <code>&lt;name&gt;@arc3</code>, where <code>&lt;name&gt;</code> <strong>SHOULD</strong> be closely related to the name in the JSON Metadata file:
<ul>
<li>If the resulting asset name can fit the <em>Asset Name</em> field, then <code>&lt;name&gt;</code> <strong>SHOULD</strong> be equal to the name in the JSON Metadata file.</li>
<li>If the resulting asset name cannot fit the <em>Asset Name</em> field, then <code>&lt;name&gt;</code> <strong>SHOULD</strong> be a reasonable shorten version of the name in the JSON Metadata file.</li>
</ul>
</li>
</ul>
</li>
<li><em>Asset URL</em> (<code>au</code>): a URI pointing to a JSON Metadata file. This URI as well as any URI in the JSON Metadata file:
<ul>
<li><strong>SHOULD</strong> be persistent and allow to download the JSON Metadata file forever.</li>
<li><strong>MAY</strong> contain the string <code>{id}</code>. If <code>{id}</code> exists in the URI, clients <strong>MUST</strong> replace this with the asset ID in decimal form. The rules below applies after such a replacement.</li>
<li><strong>MUST</strong> follow <a href="https://www.ietf.org/rfc/rfc3986.txt" rel="nofollow">RFC-3986</a> and <strong>MUST NOT</strong> contain any whitespace character</li>
<li><strong>SHOULD</strong> use one of the following URI schemes (for compatibility and security): <em>https</em> and <em>ipfs</em>:
<ul>
<li>When the file is stored on IPFS, the <code>ipfs://...</code> URI <strong>SHOULD</strong> be used. IPFS Gateway URI (such as <code>https://ipfs.io/ipfs/...</code>) <strong>SHOULD NOT</strong> be used.</li>
</ul>
</li>
<li><strong>SHOULD NOT</strong> use the following URI scheme: <em>http</em> (due to security concerns).</li>
<li><strong>MAY</strong> be a relative URI when inside the JSON Metadata file. In that case, the relative URI is relative to the Asset URL. The Asset URL <strong>SHALL NOT</strong> be relative. Relative URI <strong>MUST</strong> not contain the character <code>:</code>. Clients <strong>MUST</strong> consider a URI as relative if and only if it does not contain the character <code>:</code>.</li>
</ul>
</li>
<li><em>Asset Metadata Hash</em> (<code>am</code>):
<ul>
<li>
<p>If the JSON Metadata file specifies extra metadata <code>e</code> (property <code>extra_metadata</code>), then <code>am</code> is defined as:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="am = SHA-512/256(&quot;arc0003/am&quot; || SHA-512/256(&quot;arc0003/amj&quot; || content of JSON Metadata file) || e)
"><pre lang="plain"><code>am = SHA-512/256("arc0003/am" || SHA-512/256("arc0003/amj" || content of JSON Metadata file) || e)
</code></pre></div>
<p>where <code>||</code> denotes concatenation and SHA-512/256 is defined in <a href="https://doi.org/10.6028/NIST.FIPS.180-4" rel="nofollow">NIST FIPS 180-4</a>.
The above definition of <code>am</code> <strong>MUST</strong> be used when the property <code>extra_metadata</code> is specified, even if its value <code>e</code> is the empty string.
Python code to compute the hash and a full example are provided below (see "Sample with Extra Metadata").</p>
<blockquote>
<p>Extra metadata can be used to store data about the asset that needs to be accessed from a smart contract. The smart contract would not be able to directly read the metadata. But, if provided with the hash of the JSON Metadata file and with the extra metadata <code>e</code>, the smart contract can check that <code>e</code> is indeed valid.</p>
</blockquote>
</li>
<li>
<p>If the JSON Metadata file does not specify the property <code>extra_metadata</code>, then <code>am</code> is defined as the SHA-256 digest of the JSON Metadata file as a 32-byte string (as defined in <a href="https://doi.org/10.6028/NIST.FIPS.180-4" rel="nofollow">NIST FIPS 180-4</a>)</p>
</li>
</ul>
</li>
</ul>
<p>There are no requirements regarding the manager account of the ASA, or its the reserve account, freeze account, or clawback account.</p>
<h4><a id="user-content-pure-and-fractional-nfts" class="anchor" aria-hidden="true" href="#pure-and-fractional-nfts"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Pure and Fractional NFTs</h4>
<p>An ASA is said to be a <em>pure non-fungible token</em> (<em>pure NFT</em>) if and only if it has the following poperties:</p>
<ul>
<li><em>Total Number of Units</em> (<code>t</code>) <strong>MUST</strong> be 1.</li>
<li><em>Number of Digits after the Decimal Point</em> (<code>dc</code>) <strong>MUST</strong> be 0.</li>
</ul>
<p>An ASA is said to be a <em>fractional non-fungible token</em> (<em>fractional NFT</em>) if and only if it has the following properties:</p>
<ul>
<li><em>Total Number of Units</em> (<code>t</code>) <strong>MUST</strong> be a power of 10 larger than 1: 10, 100, 1000, ...</li>
<li><em>Number of Digits after the Decimal Point</em> (<code>dc</code>) <strong>MUST</strong> be equal to the logarithm in base 10 of total number of units.</li>
</ul>
<blockquote>
<p>In other words, the total supply of the ASA is exactly 1.</p>
</blockquote>
<h3><a id="user-content-json-metadata-file-schema" class="anchor" aria-hidden="true" href="#json-metadata-file-schema"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>JSON Metadata File Schema</h3>
<blockquote>
<p>The JSON Medata File schema follow the Ethereum Improvement Proposal <a href="https://eips.ethereum.org/EIPS/eip-1155" rel="nofollow">ERC-1155 Metadata URI JSON Schema</a> with the following main differences:</p>
<ul>
<li>Support for integrity fields for any file pointed by any URI field as well as for localized JSON Metadata files.</li>
<li>Support for mimetype fields for any file pointed by any URI field.</li>
<li>Support for extra metadata that is hashed as part of the Asset Metadata Hash (<code>am</code>) of the ASA.</li>
<li>Adding the fields <code>external_url</code>, <code>background_color</code>, <code>animation_url</code> used by <a href="https://docs.opensea.io/docs/metadata-standards" rel="nofollow">OpenSea metadata format</a>.</li>
</ul>
</blockquote>
<p>Similarly to ERC-1155, the URI does support ID substitution. If the URI contains <code>{id}</code>, clients <strong>MUST</strong> substitute it by the asset ID in <em>decimal</em>.</p>
<blockquote>
<p>Contrary to ERC-1155, the ID is represented in decimal (instead of hexadecimal) to match what current APIs and block explorers use on the Algorand blockchain.</p>
</blockquote>

<h4><a id="user-content-integrity-fields" class="anchor" aria-hidden="true" href="#integrity-fields"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Integrity Fields</h4>
<p>Compared to ERC-1155, the JSON Metadata schema allows to indicate digests of the files pointed by any URI field.
This is to ensure the integrity of all the files referenced by the ASA.
Concretly, every URI field <code>xxx</code> is allowed to have an optional associated field  <code>xxx_integrity</code> that specifies the digest of the file pointed by the URI.</p>
<p>The digests are represented as a single SHA-256 integrity metadata as defined in the <a href="https://w3c.github.io/webappsec-subresource-integrity" rel="nofollow">W3C subresource integrity specification</a>.
Details on how to generate those digests can be found on the <a href="https://developer.mozilla.org/en-US/docs/Web/Security/Subresource_Integrity" rel="nofollow">MDN Web Docs</a> (where <code>sha384</code> or <code>384</code> are to be replaced by <code>sha256</code> and <code>256</code> respectively as only SHA-256 is supported by this ARC).</p>
<p>It is <strong>RECOMMENDED</strong> to specify all the <code>xxx_integrity</code> fields of all the <code>xxx</code> URI fields, except for <code>external_url_integrity</code> when it points to a potentially mutable website.</p>
<p>Any field with a name ending with <code>_integrity</code> <strong>MUST</strong> match a corresponding field containing a URI to a file with a matching digest.
For example, if the field <code>hello_integrity</code> is specified, the field <code>hello</code> <strong>MUST</strong> exist and <strong>MUST</strong> be a URI pointing to a file with a digest equal to the digest specified by <code>hello_integrity</code>.</p>
<h4><a id="user-content-mime-type-files" class="anchor" aria-hidden="true" href="#mime-type-files"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>MIME Type Files</h4>
<p>Compared to ERC-1155, the JSON Metadata schema allows to indicate the MIME type of the files pointed by any URI field.
This is to allow clients to display appropriately the resource without having to first query it to find out the MIME type.
Concretly, every URI field <code>xxx</code> is allowed to have an optional associated field  <code>xxx_integrity</code> that specifies the digest of the file pointed by the URI.</p>
<blockquote>
<p>If MIME types are not properly specified, clients may need to rely on brittle methods to guess the MIME types such as using file extensions.</p>
</blockquote>
<p>It is <strong>RECOMMENDED</strong> to specify all the <code>xxx_mimetype</code> fields of all the <code>xxx</code> URI fields, except for <code>external_url_mimetype</code> when it points to a website.</p>
<p>Any field with a name ending with <code>_mimetype</code> <strong>MUST</strong> match a corresponding field containing a URI to a file with a matching digest.
For example, if the field <code>hello_mimetype</code> is specified, the field <code>hello</code> <strong>MUST</strong> exist and <strong>MUST</strong> be a URI pointing to a file with a digest equal to the digest specified by <code>hello_mimetype</code>.</p>
<h4><a id="user-content-localization" class="anchor" aria-hidden="true" href="#localization"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Localization</h4>
<p>If the JSON Metadata file contains a <code>localization</code> attribute, its content <strong>MAY</strong> be used to provide localized values for fields that need it. The <code>localization</code> attribute should be a sub-object with three <strong>REQUIRED</strong> attributes: <code>uri</code>, <code>default</code>, <code>locales</code>, and one <strong>RECOMMENDED</strong> attribute: <code>integrity</code>. If the string <code>{locale}</code> exists in any URI, it <strong>MUST</strong> be replaced with the chosen locale by all client software.</p>
<blockquote>
<p>Compared to ERC-1155, the <code>localization</code> attribute contains an additional optional <code>integrity</code> field that specify the digests of the localized JSON files.</p>
</blockquote>
<p>It is <strong>RECOMMENDED</strong> that <code>integrity</code> contains the digests of all the locales but the default one.</p>

<h2><a id="user-content-rationale" class="anchor" aria-hidden="true" href="#rationale"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Rationale</h2>
<p>These conventions are heavily based on Ethereum Improvement Proposal <a href="https://eips.ethereum.org/EIPS/eip-1155" rel="nofollow">ERC-1155 Metadata URI JSON Schema</a> to facilitate interoperobility.</p>
<p>The main differences are highlighted below:</p>
<ul>
<li>Asset Name and Asset Unit can be optionally specified in the ASA parameters. This is to allow wallets that are not aware of ARC-3 or that are not able to retrieve the JSON file to still display meaningful information.</li>
<li>A digest of the JSON Metadata file is included in the ASA parameters to ensure integrity of this file. This is redundant with the URI when IPFS is used. But this is important to ensure the integrity of the JSON file when IPFS is not used.</li>
<li>Similarly, the JSON Metadata schema is changed to allow to specify the SHA-256 digests of the localized versions as well as the SHA-256 digests of any file pointed by a URI property.</li>
<li>MIME type fields are added to help clients know how to display the files pointed by URI.</li>
<li>When extra metadata are provided, the Asset Metadata Hash parameter is computed using SHA-512/256 with prefix for proper domain separation. SHA-512/256 is the hash function used in Algorand in general (see the list of prefixes in <a href="https://github.com/algorand/go-algorand/blob/master/protocol/hash.go">https://github.com/algorand/go-algorand/blob/master/protocol/hash.go</a>). Domain separation is especially important in this case to avoid mixing hash of the JSON Metadata file with extra metadata. However, since SHA-512/256 is less common and since not every tool or library allows to compute SHA-512/256, when no extra metadata is specified, SHA-256 is used instead.</li>
<li>Support for relative URI is added to allow storing both the JSON Metadata files and the files it refers to in the same IPFS directory.</li>
</ul>
<p>Valid JSON Metadata files for ERC-1155 are valid JSON Metadata files for ARC-3.
However, it is highly recommended that users always include the additional RECOMMENDED fields, such as the integrity fields.</p>
<p>The asset name is either <code>arc3</code> or suffixed by <code>@arc3</code> to allow client software to know when an asset follows the conventions.</p>
